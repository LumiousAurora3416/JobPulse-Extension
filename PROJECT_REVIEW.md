## 一、背景与痛点发现

### 1.1 起因

2026 年秋招季，我开始了密集的投递。几周下来发现一个很普遍但又一直被忽视的问题：

**投递的"黑箱效应"**——简历投出去的那一刻，就是信息丢失的开始。

具体来说有几个痛点：

- **投递记录分散**：在不同公司的招聘官网、不同平台（Boss直聘、实习僧、官网投递）投了几十个岗位，没有统一的管理入口
- **跟进靠记忆力**：哪个投了几天了、哪个该跟进、哪个已经有反馈了，全靠脑子记，一周不看就忘了
- **JD 信息不可回溯**：当时投的时候看了 JD，过了一周就忘了具体要什么技能，面试时也做不了针对性准备
- **进度无法量化**：投了多少、面试转化率多少、哪个方向的命中率高，这些数据看不到，也就没法优化策略

### 1.2 初期"解决方案"（其实是个问题）

最开始我用飞书多维表格手动记录，每次投完切过来填一遍。但是：

- 手动填写效率低，容易漏填
- JD 还需要复制粘贴，很麻烦
- 填了两天就坚持不下去了

**工具本身不能增加用户的认知负担**，这是做产品最基本的原则，但我一开始自己都没做到。

## 二、需求定义与方案设计

### 2.1 核心需求

基于痛点，我定义了三个核心需求：

| 需求 | 优先级 | 描述 |
|---|---|---|
| **一键同步** | P0 | 在招聘页面点击插件，自动抓取信息写入飞书表格 |
| **投递追踪** | P1 | 对超过3天的投递自动提醒跟进，形成状态闭环 |
| **归因分析** | P2 | 分析 JD 技能要求分布，给出投递策略建议 |

### 2.2 技术方案选型

方案的核心约束：

1. **零服务器成本**（初期不想投入）
2. **数据在我手里**（飞书表格是存储层）
3. **去中心化**（不依赖第三方平台）

最终敲定的架构：

```
┌─────────────────────────────────────────────────────────┐
│                    Chrome 插件层                          │
│  企业招聘官网 → 插件抓取(岗位/公司/JD) → 飞书多维表格    │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│                    Agent 层（Python）                     │
│  每日定时查询表格 → 筛选待跟进记录 → LLM 分析            │
│  → 发送飞书交互卡片 → 按钮点击回调 → 更新表格状态       │
└─────────────────────────────────────────────────────────┘
```

**为什么选飞书生态作为基础设施？**

- 飞书多维表格 = 数据库 + UI + API，一个产品搞定
- 飞书消息卡片 = 通知 + 交互界面
- 飞书开放平台 = 认证 + 权限管理
- 我需要做的只是中间的"胶水代码"

### 2.3 飞书多维表格设计

| 字段 | 类型 | 来源 |
|---|---|---|
| 岗位 | 文本 | 插件自动抓取 |
| 公司 | 文本 | 插件自动抓取 |
| 岗位JD | 文本 | 插件自动提取（职责+要求段落） |
| 投递链接 | URL | 当前页面URL |
| 结果 | 单选 | 简历 / 面试 / 无反馈 / 简历挂 |
| 提醒状态 | 单选 | 待跟进 / 已跟进 / 已失效 / 有反馈 |
| 投递天数 | 公式 | DAYS(TODAY(), {投递时间}) |

## 三、技术实现全记录

### 3.1 Chrome 插件开发

#### 技术选型

- **Manifest V3**：Chrome 最新扩展规范
- **Vanilla JS**：纯原生 JS，无框架依赖
- **Chrome APIs**：tabs / scripting / activeTab

最开始也想过用 React 写插件，但对于一个弹窗页面来说太重了。原生 JS 足够，而且减少了构建步骤。

#### 爬虫核心逻辑：extractJobPageData()

这是插件最核心的函数，在用户点击插件时注入到页面中执行。它按照优先级从高到低的顺序尝试提取信息：

```
1. JSON-LD 结构化数据（最可靠，但很多网站没有）
2. hostname 内置公司名映射（阿里/腾讯/字节/快手）
3. CSS 选择器匹配岗位名 DOM 元素
4. document.title 评分制提取
5. h1/h2 标签提取（跳过公司名和招聘标签）
6. 公司名 DOM 选择器兜底
7. JD 正文提取 → extractDescReqModules()
```

#### 问题 1：JD 标题覆盖不全（Issue #1）

**现象**：插件只抓取"职位描述"和"职位要求"两个标题下的内容，但各公司用的标题五花八门。

**调研发现**：

| 公司 | 职责类标题 | 要求类标题 |
|---|---|---|
| 字节跳动 | 职位描述 | 职位要求 |
| 腾讯 | **工作职责** | **工作要求** |
| 美团 | **岗位职责** | **任职要求** |
| 百度 | **岗位描述** | **任职条件** |
| 快手 | 工作职责 | 任职要求 |

**解决方案**：重构 extractDescReqModules 函数，从硬编码的 2 种标题扩展到 20+ 种变体，按语义分为"职责类"和"要求类"，通过 findFirst 遍历匹配。

**代码位置**：popup.js:126-210

#### 问题 2：岗位名和公司名抓反（Issue #2）

**现象**：有些页面上岗位名和公司名填反了，公司名出现在岗位字段。

**根因**：document.title 如"字节跳动 - 产品经理 - 校园招聘"，按分隔符切分后取了第一个分段"字节跳动"作为岗位名。

**修复方案**：

```
1. title 分段改为评分制：长度 + 岗位关键词加权，选分最高的
2. 从左右取 → 改为按内容质量评分
3. 交叉验证：用其他分段自动补全公司名
4. DOM 选择器兜底提取公司名
```

**学到了什么**：提取类功能不要用"先到先得"的策略，要用"评分制"找最优解。

#### 问题 3：React SPA 页面的适配

**现象**：在快手招聘官网（zhaopin.kuaishou.cn）上，岗位名抓不到，显示为"日常实习"。

**根因**：快手招聘是 React SPA，岗位名在 `<div class="job-info-header-title-name">` 中，不是 h1/h2 标签，也没在 document.title 里。

**解决关键**：

1. 用户手动打开开发者工具告诉我 class 名称
2. 新增 `[class*="title-name"]` 选择器覆盖此类模式
3. 添加了 kuaishou 域名自动识别公司名

**产品启示**：很多时候用户能提供的信息比任何技术调研都直接。

### 3.2 飞书 Open API 集成

#### 权限体系踩坑

飞书的权限体系有三层：

| 层 | 说明 | 坑点 |
|---|---|---|
| 权限开通 | 在开发者后台勾选 | 开通后**必须发布新版本**才生效 |
| 应用能力 | 机器人 / 网页应用 | 发消息需要先启用"机器人"能力 |
| API 范围 | tenant_access_token | 不同 API 需要不同 scope |

**踩坑记录**：

1. 调用 API 返回 `[99991672] Access denied` → 开通对应权限
2. 开通权限后还是报错 → 还没发布新版本
3. 发了版本还是报错 → 机器人能力没开启
4. 机器人开了还是报错 → 需要再加 im:message:send 权限
5. 权限全了 → 终于能发卡片了

**教训**：飞书的权限管理是"三层过滤"——开通 → 发布 → 启用，少一步都不行。

### 3.3 Agent 投递追踪系统

#### 架构设计

```
agent/
├── feishu.py         # 飞书 API 客户端（查/改记录、发卡片）
├── cards.py          # 消息卡片模板
├── agent.py          # 主逻辑（日追踪 + 周分析）
├── llm_client.py     # BYOK LLM 集成
├── callback_server.py # 卡片按钮回调服务
├── config.py         # 配置
└── requirements.txt  # 依赖
```

#### 高频追踪逻辑（日）

```
每天 9:00 触发：
  1. 查飞书表格所有记录
  2. 筛选 提醒状态=待跟进 且 投递天数≥3
  3. 发送飞书跟进提醒卡片（含交互按钮）
  4. 保存 message_id 映射到本地文件
```

#### 低频归因逻辑（周）

```
每周执行：
  1. 读取所有投递记录的 JD
  2. 调用 LLM 分析技能要求分布
  3. 生成投递画像 + 策略建议
  4. 发送复盘周报卡片
```

### 3.4 交互卡片与状态闭环

这是整个项目最折腾的部分。

#### 问题 4：卡片按钮回调的响应式卡片更新

**现象**：点击按钮后飞书表格更新了，但卡片没有变成"状态已更新"。

**尝试路径**：

| 尝试 | 方法 | 结果 |
|---|---|---|
| 方案A | 在回调 HTTP 响应中返回新卡片 | 卡片不更新 |
| 方案B | Python http.server → 改为 Flask | 卡片不更新 |
| 方案C | 手动构造 JSON 响应 | 卡片不更新 |
| 方案D | 用飞书 API 更新消息（缺 message_id） | 无法调用 |
| ✅ 方案E | Agent 发卡片时记录 message_id，回调时查表调用 API | 成功 |

**根因定位**：通过加日志打印，发现回调 payload 里没有 `message_id` 字段，所以 API 更新无从谈起。

**最终方案**：Agent 发卡片时把 `record_id → message_id` 映射保存到 `message_store.json`，回调时读取映射，用飞书 API 的 PATCH 方法更新消息内容。

**编码启示**：HTTP 回调响应更新卡片可能是飞书旧版才支持的机制，新版需要用 API 全量替换。

#### 按钮功能设计

三个按钮的语义和映射逻辑：

| 按钮 | 提醒状态 | 结果 |
|---|---|---|
| ✏️ 面试 | 有反馈 | 面试 |
| 📋 无反馈 | 已跟进 | 无反馈 |
| ❌ 简历挂 | 已失效 | 简历挂 |

按钮的 confirm 弹窗设计是为了防止误触，这是典型的防错设计原则。

### 3.5 ngrok 与本地调试

回调服务需要公网地址，选择了 ngrok（之前用过）。

**架构**：

```
飞书服务器 → ngrok 隧道 → localhost:8000 → Flask 回调服务
```

**注意事项**：
- ngrok 免费版每次重启 URL 会变（需要更新飞书配置）
- Flask 开发服务器有性能警告但不影响功能
- 本地调试时三个窗口要保持运行：Agent、回调服务、ngrok

## 四、产品与技术决策记录

### 4.1 飞书原生 UI vs 自建前端看板

| 维度 | 飞书多维表格 | 自建前端 |
|---|---|---|
| 实用性 | 即开即用，在飞书内直接看 | 需要单独打开一个页面 |
| 美观度 | 固定样式，无法自定义 | 完全可控 |
| 丰富度 | 统计图表够用 | 可自由组合 |
| 实现难度 | 零开发 | 前后端都要做 |

**决策**：5 分钟搭建飞书统计视图跑通 MVP，以后有精力再自建。

### 4.2 本地 ngrok 调试 vs 云服务器

| 选择 | 成本 | 适用阶段 |
|---|---|---|
| ngrok + 本地 | 免费 | 开发调试、个人使用 |
| 云服务器（￥50/月） | 有成本 | 产品化、多用户 |

**决策**：先用 ngrok 验证产品价值，有用户后再买服务器。

### 4.3 商业化路径

产品本质不是卖工具，而是卖"秋招过程中的确定性"。

```
免费版（开源引流）:
  Chrome 插件 + 飞书同步 / 手动跟进
  → 解决"记录难"的问题

Pro 版（收费）:
  Agent 自动提醒 + 归因分析 + 周报
  → 解决"不知道投得对不对"的问题
```

用户画像：25 届/26 届秋招学生，有一定技术能力但不想折腾，愿意为效率付费。

## 五、项目复盘与经验提炼

### 5.1 做得好的

**需求验证**：先有了自己的真实痛点才开始做，不是拍脑袋想需求。

**渐进式迭代**：从最核心的"一键同步"开始，再逐步加追踪、加分析，每次只增加一个问题的复杂度。

**用户参与调试**：快手 SPA 适配那次，用户直接告诉我 class 名称，5 分钟定位问题。

**错误处理**：插件的 API 调用全部有 try-catch，用户侧有清晰的错误提示。

### 5.2 踩过的坑

**飞书的权限体系**：三层过滤（开通 + 发布 + 启用），踩了 5 次才搞清楚。

**回调 payload 没 message_id**：如果不是加日志打印，可能还在纠结为什么响应格式不对。

**CSS 选择器太宽泛**：`[class*="recruit-company"]` 可能匹配非公司名的元素。

**h1/h2 提取顺序**：先处理标题再才到 h1/h2，否则容易被公司名抢占。

### 5.3 如果重来会怎么做

1. **一开始就读飞书 API 文档的权限部分**——节省 80% 的踩坑时间
2. **先测飞书 API 的 callback payload**——确认有 message_id 再设计卡片更新方案
3. **写一个页面结构化提取器**——用一个统一的 CSS 选择器配置表，而不是在代码里硬编码

### 5.4 作为 PM 的收获

**技术决策也是产品决策**：

- 为什么选飞书？因为用户的效率工具链已经在这里了
- 为什么不做 App？因为插件在招聘场景中侵入感最小
- 为什么按钮要加 confirm？因为误触的成本远高于点击的成本

**最关键的认知转变**：

从"把功能做出来"到"让用户用完不焦虑"——追踪系统的核心价值不是记录，而是**给用户一种掌控感**。
